# Analyze Project

프로젝트 전체 맥락 파악: 구조 발견 → 카테고리별 분석 → 종합 요약

## 분석 대상 카테고리

프로젝트가 어떤 디렉토리 구조를 쓰든 다음 6개 카테고리로 매핑해서 다룹니다. 0단계에서 각 카테고리가 **이 프로젝트의 어떤 경로에 있는지** 매핑한 뒤 이후 단계를 진행합니다.

| ID    | 카테고리          | 찾는 것                                                                          |
| ----- | ----------------- | -------------------------------------------------------------------------------- |
| **A** | 리서치 자료       | 시장조사·판례·벤치마크·레퍼런스 — 기획의 "왜" 근거                               |
| **B** | 기획 문서         | PRD·요구사항·로드맵·review — 만들 "무엇" 정의                                    |
| **C** | 디자인 자료       | 화면 스펙·컴포넌트 카탈로그·토큰·디자인 파일(.pen/.fig/.sketch)                  |
| **D** | 콘텐츠 템플릿     | 카드뉴스·SNS·이메일 템플릿·에디토리얼 가이드 (프로젝트 성격에 따라 없을 수 있음) |
| **E** | 데이터 파이프라인 | 러너 스크립트·cron/launchd·시드·마이그레이션                                     |
| **F** | 구현 코드         | 애플리케이션 코드·테스트·설정                                                    |

**없는 카테고리는 건너뜀.** 예: 순수 CLI 툴이면 C·D는 없을 수 있음.

---

## 워크플로

각 단계 완료 후 **짧은 중간 요약**(2~3줄)을 사용자에게 제시하고, 마지막에 **통합 요약**을 작성합니다. 사용자가 범위를 좁히면(예: "코드만", "디자인만") 해당 카테고리만 분석합니다.

### 0. 구조 발견 (Discovery)

**목표**: 위 6개 카테고리가 이 프로젝트의 어떤 경로에 있는지 매핑.

#### 0-1. 루트 구조 훑기

```bash
ls <project-root>
# + README.md / CLAUDE.md / package.json 등 메타 파일 확인
```

#### 0-2. 흔한 패턴과 매핑

각 카테고리별로 **이런 이름이면 해당 카테고리로 매핑**:

| 카테고리                 | 흔한 디렉토리·파일 이름                                                                                                                   |
| ------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------- |
| **A. 리서치**            | `agents/reference/`, `research/`, `docs/research/`, `_workspace/input/`, `references/`, `*-research-*.md`                                 |
| **B. 기획**              | `docs/`, `specs/`, `planning/`, `_workspace/phase1/`, `_workspace/handoff/`, `prd.md`, `requirements.md`, `roadmap.md`, `project-plan.md` |
| **C. 디자인**            | `design/`, `_workspace/phase2/`, `figma/`, `*.pen`, `*.fig`, `*.sketch`, `design-final.md`, `design-spec.md`                              |
| **D. 콘텐츠**            | `_workspace/phase3/`, `templates/`, `content/`, `cards/`, `assets/templates/`, `editorial/`                                               |
| **E. 데이터 파이프라인** | `scripts/`, `cron/`, `jobs/`, `workers/`, `pipelines/`, `launchd/`, `*/cron.json`, `vercel.json` (cron 섹션)                              |
| **F. 구현 코드**         | `src/`, `app/`, `lib/`, `packages/`, `pkg/`, `internal/`, `cmd/`, `backend/`, `frontend/`                                                 |

이 매핑은 **힌트일 뿐**. 프로젝트별로 이름이 다르면 직접 추론. README/CLAUDE.md에 이미 설명되어 있을 가능성이 높으니 먼저 확인.

#### 0-3. 버전·단계 구조 감지

프로젝트가 **단계별 이터레이션** 구조를 쓰는지 확인:

- `phase{N}/v{M}/` 같은 단계-버전 중첩
- `handoff/`·`final/`·`output/` 같은 명시적 정본 디렉토리
- `v1/` `v2/` 같은 버전 분리만 있는 경우

있으면 **최신 버전(정본)만** 분석 대상. 이전 버전은 변경 이력 역추적 용도.

#### 0-4. 파일 크기 측정 (병렬)

```bash
# 카테고리별 파일 개수·크기 한 번에
wc -l <카테고리A 경로>/**/*.md <카테고리B 경로>/**/*.md ... 2>/dev/null | sort -rn | head -40
# 디자인 파일(바이너리)은 별도
find . -type f \( -name "*.pen" -o -name "*.fig" -o -name "*.sketch" \) 2>/dev/null
ls -la <찾은 디자인 파일들>  # 크기로 정본 vs 워킹 카피 판별
```

#### 0-5. 매핑 테이블 작성 및 중간 보고

```markdown
| 카테고리      | 경로                                                | 파일 수                     | 비고                      |
| ------------- | --------------------------------------------------- | --------------------------- | ------------------------- |
| A. 리서치     | agents/reference/                                   | 9개 md                      | 요약본 3개 + 풀리포트 6개 |
| B. 기획       | \_workspace/handoff/v2/, docs/{project}/            | prd-final.md 등             | v2가 최신 정본            |
| C. 디자인     | \_workspace/handoff/v2/\*.pen                       | design-legal-v2.pen (291KB) | .pen은 Pencil MCP로 접근  |
| D. 콘텐츠     | \_workspace/phase3/v2/                              | card-news-type-b.pen        | Type B 3장 확정           |
| E. 파이프라인 | scripts/brief-collect/, scripts/competitor-analyze/ | runner 2개 + util 5개       | launchd 매일 11시         |
| F. 코드       | src/                                                | Next.js 15 App Router       | -                         |
```

**중간 보고**: "카테고리 6개 중 5개 매핑 완료. D 콘텐츠 템플릿은 프로젝트 성격상 없음. 최신 정본 버전은 v2."

### 1. 카테고리별 분석

0단계 매핑 테이블을 기준으로 **있는 카테고리만** 순서대로 진행. 각 단계의 출력은 매핑된 경로에 의존.

#### A. 리서치 자료

**목표**: 프로젝트의 **이론적 기반**과 **키스톤 문서**(전체 방향을 결정한 핵심 자료) 식별.

**읽기 순서**:

1. **요약본 먼저** — `*-Summary.md`, `*-1pager.md`, 짧은 md부터
2. **키스톤 문서** — 피벗·대전환의 근거가 된 core 문서
3. **부록·링크 목록** — 제목·카테고리만 훑기 (본문 X)

**병렬 Read**: 요약본 3~5개를 한 메시지에 Read 블록으로 묶어 병렬 실행.

**산출**:

- 문서 간 의존성 그래프 (어떤 자료가 어떤 결정으로 이어졌나)
- 핵심 데이터 포인트 5~10개 (수치·판례·벤치마크)

#### B. 기획 문서

**읽기 대상**: 최신 정본 PRD만 (`handoff/v{최신}/prd-final.md` 등)

**주의**: PRD는 매우 길 수 있음(400~600줄). 다음 섹션 우선:

- 요약 / Executive Summary
- 변경 이력 (v{N} vs v{N-1}) — 피벗 여부 판단
- FR(기능 요구사항) 목록 — P0/P1 분류
- NFR·리스크 맵
- 릴리스 계획·마일스톤

**산출**:

- 핵심 피처 목록 (P0만)
- 성공 지표(North Star + Primary metrics)
- Kill Criteria 있으면 명시
- **버전 간 대전환이 있었다면** 전·후 비교 표

#### C. 디자인 자료

**읽기 대상**:

- `design-final.md` 같은 스펙 문서 (매우 길면 섹션별 부분 읽기)
- 디자인 파일(.pen/.fig/.sketch) — **전용 도구로만 접근**

**.pen 파일 워크플로** (Pencil MCP):

```
1. mcp__pencil__get_editor_state(include_schema: true)  # 처음 한 번만
2. mcp__pencil__open_document(filePath)  # 대상 파일로 전환
3. mcp__pencil__snapshot_layout(filePath, maxDepth: 0)  # top-level 프레임
4. mcp__pencil__batch_get(nodeIds, readDepth: 3)  # 핵심 프레임 상세
5. mcp__pencil__get_screenshot(nodeId)  # 대표 화면 2~3개만 시각 검증
```

**.fig/.sketch**: 해당 MCP나 API가 있으면 동일 패턴. 없으면 export된 이미지·사양서만 참고.

**정본 vs 워킹 카피**: 파일 크기·mtime으로 판별. `handoff/` 계열이 정본인 경우가 많음.

**산출**:

- Top-level 프레임(화면) 목록 + 디바이스(Desktop/Mobile)
- 재사용 컴포넌트 카탈로그
- 디자인 토큰(색·타이포·간격) 스냅샷
- 대표 화면 screenshot 2~3개

#### D. 콘텐츠 템플릿

**프로젝트에 해당 카테고리가 있을 때만**. 카드뉴스·SNS·이메일·B2B 템플릿 등.

- 바이너리 디자인 템플릿(.pen 등)은 C와 같은 워크플로
- 운영 결정 사항 확인: memory·README·CLAUDE.md에 기록된 "Type A 미사용, Type B만" 같은 판단이 있는지

**산출**: 활성 템플릿 목록 + 운영 규칙 2~3줄.

#### E. 데이터 파이프라인

**목표**: **어떤 스크립트가 어떤 데이터를 DB/파일/API에 공급하는지** 매핑.

**읽기 순서**:

1. `**/README.md` — 러너·스케줄·정책 설명
2. runner 파일만 — 이름 패턴: `run-*.ts`, `*-pipeline.*`, `backfill-*.*`, `*-analyze.*`, `cron-*.*`
3. 스케줄 설정 — `launchd/*.plist`, `vercel.json`, `crontab`, GitHub Actions `.github/workflows/`

**각 스크립트마다 추출**:

- **입력**: 어떤 테이블·파일·API에서 읽는가
- **출력**: 어디에 INSERT/UPDATE/쓰는가
- **스케줄**: 언제 실행되는가 (cron·launchd·수동·이벤트 트리거)
- **비용/제약**: LLM API·타임아웃·rate limit

**산출**: 데이터 흐름 다이어그램 (ASCII)

```
외부 소스 ──> 캐시/스테이징 ──> 메인 파이프라인 ──> DB + 캐시 invalidate
                                     │
                          별도 분석 ──> 파일 저장
```

#### F. 구현 코드

**목표**: 아키텍처 패턴 + 주요 디렉토리 맵 (전체 코드 읽기 X).

**확인 순서**:

1. **먼저 프로젝트 플랜 문서 확인** — `docs/*/01-project-plan.md`, `CLAUDE.md`, `README.md`, `ARCHITECTURE.md` 등이 기술 스택·디렉토리 구조를 이미 요약해놓았을 가능성 높음. 있으면 여기서 80% 파악.
2. 없으면 `package.json`·`go.mod`·`pyproject.toml` 등 매니페스트에서 스택 추출
3. `ls <코드 루트>` top-level만
4. 주요 하위 디렉토리(`domain/`·`adapters/`·`components/` 등) 1-depth 나열
5. **샘플 파일 1~2개만 Read**: 핵심 엔트리 포인트 상위 50줄

**아키텍처 패턴 식별**: 헥사고날 / MVC / Feature-based / Layered / Monorepo workspace 등.

**산출**:

- 디렉토리 맵 (3-depth 이내)
- 아키텍처 한 줄 요약
- 진행률: `todo.md`·이슈 트래커·마일스톤 체크박스가 있으면 비율 계산

### 2. 종합 요약

모든 단계 결과를 **하나의 정리된 맥락 문서**로 통합:

```markdown
## 제품 개요 (1~2문장)

## 대전환/피벗 이력 (있을 때)

## 현재 구현 상태

## 카테고리별 정본 경로

- A. 리서치: ...
- B. 기획: ...
- C. 디자인: ...
- D. 콘텐츠: ... (또는 "해당 없음")
- E. 파이프라인: ...
- F. 코드: ...

## 문서 간 의존성 그래프

## 다음 착수 후보
```

**원칙**:

- 전체 응답은 스크린 1~2개 분량으로 압축
- 모든 섹션 끝에 **정본 경로 링크**(파일 경로)
- 메모리에 저장할 만한 건은 `[MEMORY 후보]` 마크 (자동 저장 X, 사용자 승인 후)

---

## 핵심 원칙

- **카테고리 → 경로 순서**: 먼저 추상 카테고리를 정의하고, 0단계에서 실제 경로를 매핑. 경로를 하드코딩하지 않음
- **없는 카테고리는 건너뜀**: 6개 모두 있어야 하는 것 아님. 프로젝트 성격에 맞게 스킵
- **정본 우선**: 최신 버전만 읽고 이전은 변경 이력으로 역추적. 버전 5개 모두 읽지 않음
- **플랜 문서 먼저**: `README.md`·`CLAUDE.md`·`01-project-plan.md`가 이미 요약해놓은 내용은 재생산하지 않음
- **병렬 Read**: 독립적 파일은 한 메시지에 여러 Read/Bash 블록으로 병렬 실행
- **라인 수 먼저**: `wc -l`로 크기 측정 후 큰 파일(>500줄)은 offset/limit로 부분 읽기
- **바이너리 디자인 파일은 전용 도구만**: Read/Grep 절대 금지 (.pen=Pencil MCP 등)
- **짧은 중간 보고**: 각 단계 끝에 2~3줄 요약. 통합 요약은 마지막에만
- **범위 축소 수용**: 사용자가 "코드만" "디자인만" 요청하면 해당 카테고리만 실행

## 사용 예시

```bash
/analyze-project                    # 전체 프로젝트 파악 (0 → 있는 카테고리 모두 → 종합)
/analyze-project --scope=C          # 디자인만 (0 → C)
/analyze-project --scope=E          # 데이터 파이프라인만 (0 → E)
/analyze-project --scope=A,B        # 리서치 + 기획만 (0 → A → B)
```

**전체 실행 흐름**:

1. **0단계 Discovery**: 루트 구조·README·매핑 테이블 작성
2. 매핑된 카테고리를 A → B → C → D → E → F 순서로 분석 (없는 건 스킵)
3. 각 카테고리 끝에 2~3줄 중간 보고
4. 마지막에 통합 요약 + 다음 착수 후보 제안

## 흔한 함정

- **경로 하드코딩**: "phase1/v2/prd.md를 읽어라"처럼 고정 경로 가정하면 다른 프로젝트에서 동작 X. 항상 0단계에서 매핑 먼저
- **모든 버전 읽기**: v1·v2·v3을 전부 읽으면 시간 낭비 + 컨텍스트 오염. 최신만 + 변경 이력
- **바이너리 디자인 파일에 Read 시도**: .pen/.fig 등은 에러 발생. 반드시 전용 도구
- **스크립트 본문 전체 읽기**: README + runner 상단만으로 충분
- **코드 트리 전체 훑기**: 파일 수가 많아 컨텍스트 폭발. 플랜 문서·README가 이미 요약해놓았을 가능성 매우 높음
- **중간 보고 생략**: 한 번에 수십 파일 읽고 마지막에 한꺼번에 요약하면 사용자가 방향 수정 불가. 단계별 짧은 보고 필수
- **카테고리 D 강제 적용**: 콘텐츠 템플릿이 없는 프로젝트(CLI 툴·라이브러리 등)에 억지로 찾지 말 것
